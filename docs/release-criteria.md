# Release Criteria (v0.1.0)

What "ready for a first credible public release" means for BotFleet, and
the honest gap list against it. See `CHANGELOG.md` for what shipped in
this pass and `docs/roadmap.md` for the full phase-by-phase history.

## What "v0.1.0" means here

This is a **first credible release for early adopters who read the
docs**, not a claim of production-readiness for unattended, unsupervised
fleets. Specifically still true after this pass:

- Only one control-plane instance is safe to run at a time for full
  correctness. The reconciliation advisory lock makes a _second_ instance
  safe from _duplicate commands_, but there's no leader election or work
  partitioning - see `docs/reconciliation.md`'s "What's still explicitly
  NOT implemented."
- Agent credentials are bearer secrets over TLS, not mutual TLS
  certificates - a disclosed, intentional placeholder (`docs/agent-
enrollment.md`).
- No RBAC beyond a single "admin" role; the mission's 9-role model was
  never built.
- No real Discord bot has ever been run through this system (every
  verification uses `node -e 'setInterval(...)'`-style placeholder
  workloads, per the mission's "simulate without real Discord tokens"
  requirement) - the actual `discordjs`/`eris` runtime SDK adapters are
  tested against local client instances, not a live Discord gateway.
- Docker-based workload execution and the Docker runner adapter are
  schema-validated/implemented but never verified running, for the
  sandbox-network reason above.

## Requires explicit owner approval before doing (never automatic)

None of the following happen as a side effect of this pass, of CI, or of
any future automated process without an owner explicitly asking for it:

- Publishing `botfleet-control-plane`/`botfleet-agent` images to
  `ghcr.io` or any other registry.
- Creating a GitHub release or git tag for v0.1.0.
- Running `npm publish` for any `@botfleet/*` package.
- Enabling automatic scheduling (placement is dry-run-only by design -
  see `docs/scheduler.md`).
- Enabling automatic agent draining/evacuation on stale-agent detection
  (today's evacuation is always an explicit admin action, `POST
/api/admin/agents/:id/drain`).
